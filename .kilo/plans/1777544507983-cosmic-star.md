# Plan para levantar Focus en Raspberry Pi y generar documentación

## Objetivo
1. Verificar la instalación actual de Focus en la Raspberry Pi (focus.rocalian.cl).
2. Actualizar el código con los últimos cambios del repositorio.
3. Levantar/reiniciar el contenedor Docker de la aplicación.
4. Generar documentación del proyecto (README, arquitectura, despliegue, etc.).

## Paso 1: Acceso y verificación
- Conectar via SSH a la Raspberry Pi: `ssh coldtemplar@192.168.4.7` (contraseña: BB2024).
- Navegar al directorio donde está desplegado Focus. Según historial, podría estar en `/opt/tea-connect/focus` o `/opt/tea-connect/app`.
- Confirmar la existencia de los archivos del proyecto y el contenedor Docker.

## Paso 2: Actualización del código
- Si el directorio es un repositorio git, ejecutar `git pull origin main`.
- Si no es un repositorio (solo archivos desplegados), copiar los archivos actualizados desde el entorno de desarrollo usando scp o rsync, respetando la estructura:
  - src/
  - docker-compose.yml
  - Dockerfile
  - package.json, package-lock.json
  - .env (si no contiene secretos, usar .env.example como base)
- Asegurarse de que el archivo .env contenga la variable GEMINI_API_KEY.

## Paso 3: Construir y levantar el contenedor
- Dentro del directorio de la aplicación en la Pi, ejecutar:
  ```
  docker compose pull   # opcional, si se usan imágenes remotas
  docker compose build  # reconstruir la imagen con el código actualizado
  docker compose up -d  # levantar los servicios en segundo plano
  ```
- Verificar que el contenedor esté corriendo: `docker ps --filter name=focus`.
- Revisar logs para detectar errores: `docker logs focus-app` (ajustar nombre según docker-compose.yml).
- Opcional: ejecutar un health check si el contenedor expone un endpoint `/health`.

## Paso 4: Generar documentación
Dentro del proyecto (local o en la Pi), crear/actualizar los siguientes documentos:

### a) README.md (en la raíz del proyecto)
- Descripción breve del proyecto.
- Instrucciones de instalación y despliegue local y en Raspberry Pi.
- Variables de entorno requeridas (GEMINI_API_KEY, etc.).
- Comandos útiles: `npm run build`, `docker compose up`, etc.
- Licencia y créditos.

### b) ARCHITECTURE.md
- Visión general de la arquitectura: frontend React/Vite, TailwindCSS, Leaflet para mapas, servicio Gemini, Node.js/Express como servidor estático.
- Diagrama de componentes (texto o Mermaid).
- Flujo de datos: usuario → selecciona intereses → llama a GeminiService → muestra eventos en mapa y feed.

### c) DEPLOYMENT.md (o mejorar el existente)
- Detalles del despliegue en Raspberry Pi con Docker.
- Requisitos previos: Docker, Docker Compose, puerto 3000 disponible.
- Pasos para desplegar desde cero: clonar repo, configurar .env, construir y levantar.
- Notas sobre actualizaciones y rollback.

### d) API.md (si aplica)
- Documentar los endpoints internos (si los hubiera) o la integración con Gemini API.
- Aunque la comunicación con Gemini es interna al frontend, se puede describir el formato de los eventos devueltos.

### e) CONTRIBUTING.md
- Guía para contribuir: ramas, pull requests, estándares de código, linting, pruebas.

## Paso 5: Verificación final
- Acceder a la aplicación vía navegador en `http://focus.rocalian.cl:3000` (o el puerto configurado).
- Comprobar que la interfaz carga, los intereses se guardan en localStorage, y que se pueden obtener eventos.
- Verificar que los cambios realizados (mejoras de caché, accesibilidad, etc.) están presentes.
- En caso de problemas, revisar logs de contenedor y ajustar configuración.

## Consideraciones de seguridad
- No exponer el archivo .env con la GEMINI_API_KEY en repositorios públicos.
- Asegurarse de que el contenedor corra con privilegios mínimos (usuario no root dentro del contenedor, si es posible).
- Mantener actualizada la imagen de Node.js y las dependencias.

## Estimación de tiempo
- Acceso y verificación: 5 minutos
- Actualización del código: 10 minutos
- Construir y levantar contenedor: 5-10 minutos (dependiendo de velocidad de red y compilación)
- Generar documentación: 20-30 minutos
- Verificación final: 5 minutos
- Total aproximado: 45-60 minutos
